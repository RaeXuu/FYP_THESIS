## Chapter 1：Introduction（引言）

### 1.1 研究背景与动机
- 心血管疾病（CVD）是全球第一死亡原因，每年约造成 1790 万人死亡（WHO数据）。
- 心音听诊是已有 200 年历史的非侵入性初步诊断方法，但传统听诊器依赖医生主观经验，无法记录数据，不支持远程监测。
- 商业数字听诊器（如 Eko Core，售价超 $300）成本高、操作复杂，限制了在资源匮乏场景中的应用。
- 本项目旨在开发低成本、便携、易用的数字心音听诊器，实现实时采集、无线传输和定量分析。

### 1.2 问题陈述与研究目标
当前解决方案存在三大不足：
1. **硬件成本高、设计复杂**：多芯片架构（音频编解码器 + MCU + 蓝牙模块），BOM 成本高。
2. **无线传输实时性差**：基于轮询的蓝牙传输延迟高，数据丢失严重。
3. **缺乏集成化信号处理工具**：大多数低成本方案只提供原始数据采集，无滤波、特征提取和可视化功能。

**研究目标：**
- 设计低功耗、微型化的硬件系统；
- 有效预处理心音信号、提取特征；
- 构建适合边缘计算的轻量级智能诊断模型；
- 实现从信号采集到结果展示的完整闭环。

### 1.3 研究方法
- **模块化设计**：分为 ESP32 采集终端、BLE 无线传输链路、Raspberry Pi 边缘处理与智能诊断平台三个模块。
- **硬件**：单芯片 ESP32-C3 方案，集成高保真音频采集和 BLE 无线传输。
- **固件**：Arduino 框架，实现 I2S 音频驱动、BLE 数据封装、Notify 模式低延迟主动推送。
- **信号处理**：三阶高通 + 二阶 Butterworth 带通滤波器 + 中值滤波算法去噪。
- **智能诊断**：轻量级 CNN-LSTM 深度学习模型，部署于 Raspberry Pi，在公开临床心音数据集上训练，实现正常/异常心音分类。

## Chapter 2：Theoretical Background and Literature Review（理论背景与文献综述）

### 2.1 心音信号的生理基础
- 心音是心脏瓣膜振动、心肌和血流在心动周期中产生的声学信号。
- **S1（第一心音）**：心室收缩开始时，房室瓣关闭产生，频率范围 20–150 Hz。
- **S2（第二心音）**：心室舒张开始时，半月瓣关闭产生，频率范围与 S1 相近。
- 有效频率范围：20–600 Hz，临床最关键信息在 20–150 Hz。
- 采样率选用 **8000 Hz**，满足奈奎斯特定理，同时最小化 BLE 数据量。

### 2.2 数字听诊器技术研究现状

#### 2.2.1 医疗设备设计
- **商业产品**：
  - **Eko Core**（Eko Devices, USA）：支持心音 + ECG 同步采集、40x 声音放大，但售价超 $300，普及受限。
  - **Apple Watch**：集成 ECG 电极和心率监测，但不是专用听诊器，不支持 PCG（心音图）信号采集。
- **学术研究**：Wang 等人（中科院）用 STM32F103RET6 开发了便携式听诊器，但采用分立硬件架构（独立 DSP 滤波 + TPA6100 放大 + 独立蓝牙模块），复杂度高、成本高。

#### 2.2.2 诊断算法演进
- **传统机器学习阶段**：基于 MFCC、STFT 等特征工程 + SVM/KNN/随机森林分类，泛化能力弱。
- **深度学习阶段**：CNN 成为处理心音频谱的主流选择；CNN-LSTM（CRNN）混合架构是研究热点。
- **边缘 AI 阶段**：Zhang 等人（2023）基于 Raspberry Pi Zero 2W（硬件成本仅约 $25），实现 CNN + 随机森林的实时检测，验证了边缘计算架构的可行性。

### 2.3 核心技术原理

#### 2.3.1 I2S 音频总线协议
- I2S 是专为数字音频设计的串行同步通信协议，使用三条信号线：
  - **SCK（串行时钟）**：位时钟
  - **WS（字选择）**：帧同步，区分左右声道
  - **SD（串行数据）**：以二进制补码格式传输 PCM 音频样本
- 本项目中 ESP32-C3 作为 I2S 主设备，驱动 INMP441 MEMS 麦克风（I2S 从设备）。

#### 2.3.2 BLE 低功耗通信协议
- 选用 **BLE 5.0**，2 Mbps PHY 层，平衡能效与吞吐量。
- 采用 **GAP + GATT 架构**：ESP32 作为 Peripheral（外设），Raspberry Pi 作为 Central（中央设备）。
- 核心传输机制：**Notify 模式**（主动推送），消除请求-响应握手的延迟，保证实时音频的时序完整性。

2.3.3 Relevant Algorithm Theories

### 2.4 系统整体框架
系统由三部分组成：
1. **硬件采集终端**：MEMS 麦克风 → ESP32-C3（I2S 采集 + 降采样 + BLE 5.0 Notify 上传）
2. **接收处理端（Raspberry Pi）**：BLE 数据接收 → 数字信号处理（滤波、归一化）
3. **自动诊断算法**：预训练 AI 深度学习模型，实时分析 → 结果存储本地数据库

## Chapter 3：System Hardware Design and Implementation（硬件设计与实现）

### 3.1 硬件系统设计演进

#### 3.1.1 初始方案：nRF52832 + LMD2718T
- 初始设计以 nRF52832 BLE SoC 和 LMD2718T PDM 麦克风为核心，目标是超低功耗和紧凑体积。

#### 3.1.2 初始方案的关键瓶颈
遇到三大无法克服的技术瓶颈：
1. **协议与接口冲突**：LMD2718T 使用 PDM 接口，nRF52832 进行软件 PDM 解码时 CPU 占用率接近 100%，导致频繁断连和系统冻结。
2. **传输带宽不足**：UART 最高波特率 115200 bps，无法满足 8kHz 16-bit 连续音频流（需 128 kbps）。
3. **实时稳定性与数据完整性风险**：CPU 竞争导致 BLE 协议栈连接维护出现问题，缓冲区溢出和数据丢失风险高。

### 3.2 核心主控模块选型
对比 nRF52832、STM32WB55CG、ESP32-C3 三款芯片，最终选择 **ESP32-C3-WROOM-02**，理由：
- 集成硬件 I2S 控制器 + 原生 DMA 支持，CPU 干预最小化；
- 160MHz RISC-V 内核，400KB SRAM，确保多任务并发调度；
- 原生 BLE 5.0 Notify 模式，最高吞吐量 1280 kbps（心音流需求的 10 倍）；
- 成本仅为 nRF52832 的 1/3、STM32WB55CG 的 1/4；
- 拥有最丰富的开源驱动资源，开发周期缩短 60% 以上。

### 3.3 心音采集模块选型
对比 ECM 麦克风、压电传感器、LMD2718T（PDM）、INMP441（I2S）四种方案，选择 **INMP441**，理由：
- 原生 I2S 数字输出，直接与 ESP32-C3 接口，无需额外 PDM 解码或外部 ADC；
- 高 SNR（61 dB），低频频响平坦，信号在传感器头部即被数字化，免疫电源噪声和 EMI；
- 封装紧凑（4.72 × 3.76 × 1 mm），低功耗，适配便携设计。

### 3.4 硬件调试与验证
自定义 PCB 包含以下核心模块：

| 模块 | 设计要点 |
|------|---------|
| **电源管理模块** | USB-C 供电，LDO 降压至 3.3V，多级去耦电容抑制 BLE 射频产生的高频纹波 |
| **MCU 最小系统** | ESP32-C3 核心电路，含复位键（SW2）和 Boot 模式键（SW3），宽频并联去耦电容防止 BLE 发射峰值导致掉电 |
| **I2S 数字音频采集接口** | INMP441 通过 SCK(IO2)/WS(IO5)/SD(IO4) 接 MCU，L/R 接 GND（左声道输出），CHIPEN 接 3.3V |
| **USB-C 编程接口** | CC1/CC2 各接 5.1kΩ 下拉电阻，USB D+/D- 直接接 ESP32-C3 内置 USB-Serial JTAG，无需外部 CP2102/CH340 芯片 |

## Chapter 4：System Software Design and Implementation（软件设计与实现）

### 4.1 整体软件架构
- **ESP32 固件（下位机）**：Arduino 框架，三核心模块：I2S 硬件驱动、心音信号预处理（去 DC、数字增益）、实时 BLE 数据传输。
- **Python 上位机**：四核心模块：BLE 数据接收、数据存储、数字信号处理、可视化。

### 4.2 ESP32 固件设计

#### 4.2.1 系统初始化
- 串口波特率 115200（调试用），GPIO 初始化：SCK→GPIO4，WS→GPIO6，SD→GPIO5。

#### 4.2.2 I2S 音频驱动配置
| 参数 | 配置值 | 设计依据 |
|------|--------|---------|
| 通信模式 | 标准 I2S 主模式 | ESP32-C3 提供时钟，确保与 INMP441 稳定同步 |
| 采样频率 | 8000 Hz | 覆盖 20–220 Hz 有效心音频段，减少 BLE 数据量 |
| 数据格式 | 16-bit PCM | 适配 BLE MTU（每包 20 字节），优化带宽利用率 |
| DMA 缓冲区 | 8 × 128 字节 | 多级缓冲防止数据覆盖与丢失 |

#### 4.2.3 BLE 通信模块设计
- 设备名称：`ESP32_Steth`
- 自定义服务 UUID：`4fafc201-1fb5-459e-8fcc-c5c9c331914b`
- 音频特征 UUID：`beb5483e-36e1-4688-b7f5-ea07361b26a8`，启用 **Notify 属性**
- 工作流程：初始化 → 创建 GATT 服务 → 添加 Notify 特征 → 开始广播 → 主机订阅后进入数据传输循环。

#### 4.2.4 信号预处理与主循环
主循环中实时进行 DSP 处理：
1. **DC 偏移去除**：滑动平均算法动态去除硬件电路产生的直流分量。
2. **数字增益**：信号放大 30 倍，增强心音波形显示灵敏度。
3. **限幅保护**：限制信号幅度在 [-32000, 32000]，防止溢出产生噪声。
4. **数据打包**：处理后的 16-bit 数据推入缓冲区，通过 Notify 方式实时发送。

#### 4.2.5 固件功能验证
- **I2S 采集验证**：通过 Arduino IDE 串口绘图仪实时监测，心音波形清晰可见。
- **BLE 传输验证**：使用 nRF Connect 手机应用，10 分钟连续测试无断连、无丢包，成功接收连续 16 进制 PCM 音频数据包。

### 4.3 信号预处理算法实现与比较

#### 4.3.1 预处理目标
去除呼吸基线漂移、环境高频噪声和电磁干扰，提取高纯度 S1/S2 特征供 CNN 模型识别。

#### 4.3.2 脉冲干扰消除——中值滤波
- 采用非线性**中值滤波**（Tukey, 1971），用窗口内中值替代中心采样值，消除脉冲尖峰同时保留边缘特征。
- 核心参数：半窗长 k=2，总窗口长度 5 个采样点（0.625 ms），兼顾脉冲消除能力与 S1 边缘保真度。

#### 4.3.3 基线漂移抑制
- 设计**三阶 Butterworth 高通滤波器**，截止频率 4 Hz，抑制呼吸运动引起的超低频基线波动。

#### 4.3.4 滤波算法对比：级联 IIR vs. 小波去噪
对 20 组实际人体心音样本（10 组静息 + 10 组运动后）进行测试：

| 处理方案 | SNR (dB) | 频谱纯净度 (dB) |
|---------|---------|--------------|
| 原始信号 | — | -7.83 |
| **级联 IIR 滤波** | **18.92** | **1.80** |
| 小波去噪 | 18.93 | 1.22 |

- 小波去噪时域静默期更平滑，但软阈值机制导致带内有效心音能量损失，频谱纯净度低于级联 IIR。
- 最终选择：**4 阶 Butterworth 带通滤波（20–220 Hz）+ 零相位滤波（FILT-FILT）**，计算开销极低、信号保真度高、无相位失真。

## 总体技术路线图

```
心音采集（INMP441）
    ↓ I2S 数字接口
ESP32-C3（8kHz 采样 + 中值滤波 + 增益 + BLE 5.0 Notify）
    ↓ BLE 无线传输
Raspberry Pi 上位机
    ├── 高通滤波（4Hz 截止，去基线漂移）
    ├── 带通滤波（20–220Hz Butterworth，去噪）
    └── CNN-LSTM 模型推理
         ↓
    正常 / 异常心音分类结果 → 存储本地数据库
```
